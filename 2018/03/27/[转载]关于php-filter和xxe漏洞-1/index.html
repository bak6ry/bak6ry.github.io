关于php://filter和xxe漏洞

发表于 2018-03-27 | 分类于 Web安全

前言

最近在刷题的时候又遇到了php://filter这个协议，想着之前也遇到过许多次，就想着做一次总结，把这个问题搞懂，也是第一次用MD来写博客，就做的慢一点，熟悉一下MD的语法，也算是给自己的博客生涯开一个好头吧！

关于php://filter协议

php://filter协议是php语言中的一个特有的协议流，它的作用是作为一个中间流来处理其他流。比如我们可以用一行代码将post请求内容转换成base64编码输出

XXE中的使用

php://filter最常出现的地方就是xxe漏洞。犹豫xxe漏洞的特殊性，我们在读取html，php等文件时可能出现此类错误parser error : StartTag: invalid element name。其原因就是，PHP是给予标记语言，<?php … ?>这个语法也与xml相符合，所以在解析xml的时候会误认为时xml，而其中（比如特殊字符）又有可能和标准xml冲突，所以导致了出错。
那么，为了读取包含有敏感信息的PHP等源文件，我们就要先将“可能引发冲突的PHP代码”编码一遍，这里就会用到php://filter。
php://filter/read=convert.base64-encode/resource=./xxe.php

巧用编码和解码

content = '<?php exit; ?>';    
$content .= $_POST['txt'];   
file_put_contents($_POST['filename'], $content);

在实战中经常会遇到在缓存、配置文件等等地方时，不允许用户直接访问的文件，都会被加上if(!defined(xxx))exit;之类的限制，则可以用php://filter流中的base64-decode方法，将$content解码，利用base64_decode函数特性去除“死亡exit”。其实base64_decode可以理解为以下代码
<?php
$_GET[‘txt’] = preg_replace(‘|[^a-z0-9A-Z+/]|s’, ‘’, $_Get[‘txt’]);
base64_decode($_GET[‘txt’]);
?>
php://filter/write=convert.base64-decode 来首先对其解码。在解码的过程中，字符<、?、;、>、空格等一共有7个字符不符合base64编码的字符范围将被忽略，所以最终被解码的字符仅有“phpexit”和我们传入的其他字符。“phpexit”一共7个字符，因为base64算法解码时是4个byte一组，所以给他增加1个“a”一共8个字符。这样，”phpexita”被正常解码，而后面我们传入的webshell的base64内容也被正常解码。结果就是<?php exit; ?>没有了。

内容来自于https://www.leavesongs.com/PENETRATION/php-filter-magic.html

XXE漏洞

原理

如果在产品功能项目设计中，用来解析的xml是外部可控的，则有可能能形成文件读取，dos，csrf漏洞。各种语言都有支持xml的组件及函数。在PHP中默认处理的函数为: xml_parse和simplexml_load。 xml_parse的实现方式为expat库，默认情况不会解析外部实体,simplexml_load默认情况下会解析外部实体,造成安全威胁(xxe漏洞无法复现的原因就是在新版本中，已经不再默认解析实体了，需要在simplexml_load_file函数中指定第三个参数为LIBXML_NOENT，不然不会解析实体的)

危害

读取任意文件
执行系统命令
探测内网端口
攻击内网网站
详细操作参考：https://www.waitalone.cn/xxe-attack.html
